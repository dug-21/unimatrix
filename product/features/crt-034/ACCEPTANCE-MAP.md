# crt-034 Acceptance Criteria Map

| AC-ID | Description | Verification Method | Verification Detail | Status |
|-------|-------------|--------------------|--------------------|--------|
| AC-01 | A `co_access` pair with `count >= CO_ACCESS_GRAPH_MIN_COUNT` that has no `CoAccess` edge in `GRAPH_EDGES` is promoted on the next background tick after the threshold is crossed | test | Unit test: insert qualifying pair into `co_access`, call `run_co_access_promotion_tick`, assert matching row exists in `graph_edges` with `relation_type = 'CoAccess'` | PENDING |
| AC-02 | A `co_access` pair already in `GRAPH_EDGES` as `CoAccess` whose normalized weight has drifted by more than `CO_ACCESS_WEIGHT_UPDATE_DELTA` (0.1) has its `weight` updated within one tick | test | Unit test: pre-insert `CoAccess` edge with stale weight (delta > 0.1), run promotion tick, assert `weight` column is updated to new normalized value | PENDING |
| AC-03 | A `co_access` pair already in `GRAPH_EDGES` as `CoAccess` whose normalized weight delta is <= 0.1 produces zero database writes | test | Unit test: pre-insert `CoAccess` edge with current weight (delta <= 0.1), run tick, assert `rows_affected == 0` on the UPDATE path and weight is unchanged | PENDING |
| AC-04 | When qualifying pairs exceed `max_co_access_promotion_per_tick`, only the top-N pairs by `count DESC` are processed; lower-count pairs are deferred | test | Unit test: seed more pairs than cap (e.g. 10 pairs, cap=3), assert only the 3 highest-count pairs appear in `GRAPH_EDGES` after one tick | PENDING |
| AC-05 | The promotion step executes after `maintenance_tick` (cleanup_stale_co_access) and orphaned-edge compaction, and before `TypedGraphState::rebuild()` in `run_single_tick` | manual | Code review of `background.rs`: confirm `run_co_access_promotion_tick` call appears between orphaned-edge compaction block and `TypedGraphState::rebuild()` call, with ORDERING INVARIANT anchor comment | PENDING |
| AC-06 | `InferenceConfig.max_co_access_promotion_per_tick` has serde default 200, validates to range [1, 10000], and participates in project-level config merge with same override semantics as `max_graph_inference_per_tick` | test | Unit tests in `config.rs`: (a) default deserializes to 200, (b) value 0 returns validation error, (c) value 10001 returns validation error, (d) project-level override 50 wins over global 200 | PENDING |
| AC-07 | `CO_ACCESS_GRAPH_MIN_COUNT: i64 = 3` is exported as a public constant from `unimatrix-store` and is used by the promotion tick | test | Unit test or doc test: assert `unimatrix_store::CO_ACCESS_GRAPH_MIN_COUNT == 3` and is accessible; grep confirms tick's batch query uses the constant, not a bare literal `3` | PENDING |
| AC-08 | `EDGE_SOURCE_CO_ACCESS: &str = "co_access"` is exported from `unimatrix-store` alongside `EDGE_SOURCE_NLI` | test | Unit test or doc test: assert `unimatrix_store::EDGE_SOURCE_CO_ACCESS == "co_access"` | PENDING |
| AC-09 | When `co_access` is empty or no pairs meet `CO_ACCESS_GRAPH_MIN_COUNT`, the promotion tick completes without error, without warning, and logs 0 inserted / 0 updated | test | Unit tests: (a) run against empty `co_access` table — assert no panic, no `warn!`, `info!` shows 0/0; (b) run against table with all counts below threshold — same assertions | PENDING |
| AC-10 | `InferenceConfig::validate()` rejects `max_co_access_promotion_per_tick = 0` with a clear error message identifying the field name and valid range | test | Unit test: assert validation error for value 0 contains the string "max_co_access_promotion_per_tick" | PENDING |
| AC-11 | When a single `write_pool_server()` call fails (simulated timeout or constraint error), `run_co_access_promotion_tick` logs at `warn!`, continues processing remaining pairs, and returns `()` without panicking | test | Unit test with injected write failure: assert function returns `()`, `warn!` emitted with pair identifiers, remaining pairs are attempted, final `info!` log appears | PENDING |
| AC-12 | All rows inserted by the promotion tick have `bootstrap_only = 0`, `source = EDGE_SOURCE_CO_ACCESS`, `created_by = 'tick'`, `relation_type = 'CoAccess'` | test | Unit test: promote one pair, SELECT the resulting row, assert all four metadata columns match expected values | PENDING |
| AC-13 | Weight normalization uses `MAX(count)` computed over ALL qualifying pairs (count >= threshold), not only the capped batch | test | Unit test: seed 10 pairs with counts [1..10], set cap=3, assert promoted pairs use `max_count=10` (global), not the max within the top-3 batch | PENDING |
| AC-14 | An existing `CoAccess` edge with `INSERT OR IGNORE` applied a second time (pair already promoted, count unchanged) produces zero rows affected and no spurious UPDATE | test | Unit test: run promotion tick twice against same qualifying pair with unchanged `co_access` count, assert exactly one `graph_edges` row exists and `weight` is unchanged | PENDING |
| AC-15 | A `co_access` pair whose count has fallen below `CO_ACCESS_GRAPH_MIN_COUNT` after promotion is NOT removed from `GRAPH_EDGES` by the promotion tick | test | Unit test: promote a pair, set its count below threshold, re-run tick, assert `GRAPH_EDGES` row still present | PENDING |
