# nan-007 Acceptance Criteria Map

| AC-ID | Description | Verification Method | Verification Detail | Status |
|-------|-------------|--------------------|--------------------|--------|
| AC-01 | `unimatrix snapshot --out <path>` creates a valid SQLite file containing all database tables (entries, query_log, graph_edges, co_access, sessions, shadow_evaluations, and all others present in the schema) | shell | `sqlite3 <path> "SELECT name FROM sqlite_master WHERE type='table'"` — assert all expected table names are present | PENDING |
| AC-02 | `unimatrix snapshot` refuses with a non-zero exit code and an error message when `--out` resolves (via `canonicalize`) to the same file as the active daemon's database | shell | `unimatrix snapshot --out <live-db-path>`; assert exit code != 0 and stderr contains both resolved paths; also test with symlink pointing to live DB | PENDING |
| AC-03 | `unimatrix eval scenarios --db <snapshot>` produces JSONL where each line is a valid scenario object with `id`, `query`, `context`, `baseline.entry_ids`, `baseline.scores`, and `source` fields | test | Parse each line as JSON; assert all required fields present with correct types; assert `len(baseline.entry_ids) == len(baseline.scores)` for every line | PENDING |
| AC-04 | `unimatrix eval scenarios --retrieval-mode uds` returns only `source="uds"` scenarios; `--retrieval-mode mcp` returns only `source="mcp"`; `--retrieval-mode all` returns both | shell | Run each flag variant against a snapshot with known query_log rows of both source types; assert every output line's `source` field matches the filter | PENDING |
| AC-05 | `unimatrix eval run --db <snapshot> --scenarios <file> --configs baseline.toml,candidate.toml` opens snapshot in read-only mode and produces no writes to the snapshot | shell | Record SHA-256 hash of snapshot before `eval run`; assert hash is byte-for-byte unchanged after `eval run` completes | PENDING |
| AC-06 | `unimatrix eval run` produces one JSON result file per scenario in `--out` containing `profiles`, `comparison.kendall_tau`, `comparison.mrr_delta`, `comparison.p_at_k_delta`, and `comparison.latency_overhead_ms` | test | Parse each result JSON; assert all required fields are present and numeric | PENDING |
| AC-07 | `unimatrix eval run` uses `baseline.entry_ids` as soft ground truth for P@K on query-log-sourced scenarios (`expected = null`), and `expected` as hard labels for hand-authored scenarios | test | Run with a hand-authored scenario (`expected = [id1, id2]`) — assert P@K computed against `expected`; run with a query-log scenario (`expected = null`, `baseline.entry_ids = [id3, id4]`) — assert P@K computed against `baseline.entry_ids` | PENDING |
| AC-08 | `unimatrix eval report --results <dir>` produces a Markdown file with a summary table, notable ranking changes section, latency distribution section, entry-level analysis section, and zero-regression check section | grep | Assert output file contains all five section headers: `## Summary`, `## Notable Ranking Changes`, `## Latency Distribution`, `## Entry-Level Analysis`, `## Zero-Regression Check` (or equivalent) | PENDING |
| AC-09 | Zero-regression check lists all scenarios where the candidate profile has lower MRR OR lower P@K than baseline; contains an explicit empty-list indicator when no regressions exist | test | Run with results where one scenario has lower MRR but equal P@K — assert that scenario appears in zero-regression list; run with no degradation — assert explicit empty-list indicator is present | PENDING |
| AC-10 | `UnimatrixUdsClient` connects to a running daemon's MCP UDS socket, completes the MCP `initialize` handshake, and executes all 12 `context_*` tool methods with the same results as `UnimatrixClient` for equivalent queries | test | `test_eval_uds.py` — start daemon with `daemon_server` fixture; run a query via `UnimatrixUdsClient`; run same query via `UnimatrixClient` stdio; assert results are identical | PENDING |
| AC-11 | `UnimatrixUdsClient` supports the context manager protocol (`__enter__` / `__exit__`) with automatic connect/initialize/disconnect lifecycle | test | `test_eval_uds.py` — use `with UnimatrixUdsClient(...) as client:` block; assert connection established on entry and closed on exit without explicit `connect()` / `disconnect()` calls | PENDING |
| AC-12 | `UnimatrixHookClient` connects to the daemon's hook socket, sends a `Ping` request, and receives a `Pong` response within the configured timeout | test | `test_eval_hooks.py` — `hook_client.ping()` returns a `HookResponse` with `type = "Pong"` | PENDING |
| AC-13 | `UnimatrixHookClient.session_start()` followed by `session_stop()` results in a session record visible in the database | test | `test_eval_hooks.py` — call `session_start` then `session_stop` via hook client; call `context_status` via `UnimatrixUdsClient`; assert the session appears in status output | PENDING |
| AC-14 | `UnimatrixHookClient` rejects payloads exceeding `MAX_PAYLOAD_SIZE` (1 MiB) with a descriptive `ValueError` before sending | test | `test_eval_hooks.py` — construct a payload of 1,048,577 bytes; assert `ValueError` raised before any bytes are sent on the socket; confirm client remains usable for a subsequent `ping()` | PENDING |
| AC-15 | `unimatrix snapshot`, `unimatrix eval scenarios`, `unimatrix eval run`, and `unimatrix eval report` are all registered as subcommands visible in `--help` output | shell | `unimatrix --help` contains `snapshot`; `unimatrix eval --help` contains `scenarios`, `run`, and `report` | PENDING |
| AC-16 | `unimatrix eval run --db <active-db>` returns a non-zero exit code with a descriptive error message when `--db` resolves (via `canonicalize`) to the same file as the active daemon's database | shell | Invoke `unimatrix eval run --db <live-db-path> --scenarios <file> --configs <profile> --out <dir>`; assert exit code != 0 and stderr contains a message naming both resolved paths (FR-44) | PENDING |
