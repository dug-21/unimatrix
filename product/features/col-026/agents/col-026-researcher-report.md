## Key Scope Decisions Made

### 1. Phase Timeline is in scope (medium effort, highest value)

The `cycle_events` query is already running in the handler (step 10g, crt-025 phase narrative).
All timestamps needed for per-phase observation slicing are already in memory. No new DB query is
required for the observation-side computation — it's an in-memory filter over `attributed` records
using the same time windows col-024 already defines. The new `PhaseStats` type and computation step
adds scope, but the data infrastructure is already wired. This is the right call.

### 2. Per-phase hotspot scoping is in scope

Annotating each finding with the phase it fired in is a formatter change (read phase windows from
`phase_stats`, compare finding evidence timestamps to windows). The underlying data exists once
`PhaseStats` is computed. This was classified as medium effort in FINDINGS.md §8 — confirmed
feasible within one feature.

### 3. Knowledge reuse fix (#320) is in scope, implementation path is clear

`compute_knowledge_reuse` in `knowledge_reuse.rs` already collects all distinct delivered entries.
The bug is that there is no feature-cycle split on those entries. The fix: in
`compute_knowledge_reuse_for_sessions`, pre-fetch each entry's `feature_cycle` field alongside its
`category` (the `get(entry_id)` call that already runs at line 2168 returns the full `EntryRecord`
including `feature_cycle`). Add a second closure or extend the first to return `(category,
feature_cycle)`. Add `cross_feature_reuse` and `intra_cycle_reuse` fields to
`FeatureKnowledgeReuse`. No schema change; no new DB query.

### 4. "What Went Well" section is in scope (formatter only)

`baseline_comparison` is already computed and passed to the formatter. Currently only
`Outlier`/`NewSignal` entries are rendered. Non-outlier entries where the current value is
favorably different from the mean (lower-is-better metrics below mean, higher-is-better above mean)
are simply not rendered. The "What Went Well" section is a new filter pass on the existing data. The
design phase must define which metrics are "lower is better" vs "higher is better."

### 5. Goal + attribution path are in scope (trivial)

`get_cycle_start_goal(cycle_id)` is a shipped DB method (v16). It is called in the UDS listener but
NOT in the `context_cycle_review` handler. Adding it is one async call + one new `Option<String>`
field on `RetrospectiveReport`. Attribution path tracking requires detecting which of the three paths
returned non-empty results in the existing three-path closure.

### 6. Threshold language replacement is in scope (formatter audit)

The `HotspotFinding.threshold: f64` field is used internally for detection logic and is on every
finding. Several claim strings in detection rules print the raw threshold value. The formatter also
renders evidence with raw `ts=` values. Both need an audit-and-replace pass. The `threshold` field
stays on the struct — it must not appear in user-facing output as a configured limit.

### 7. Rework per-pass diff is explicitly deferred

The Phase Timeline table will show pass count and per-pass duration/records via annotation rows.
The full per-pass diff (what changed between pass 1 and pass 2) requires a higher-effort design on
what "diff" means for observation data. Deferred to a follow-on feature.

### 8. Goal-contextualized hotspot severity is deferred

Inferring CycleType and suppressing expected hotspots is a design problem that goes beyond
formatting — it requires defining expected hotspot profiles per cycle type, which is a follow-on
design decision. The goal is shown in the header; it does not affect hotspot severity in col-026.

### 9. Permission-friction AC-19 is scoped as a conditional fix

After reading the detection rule, the `PermissionRetriesRule` claim string correctly says
"permission retries (Pre-Post differential)" — not "permission prompts." The false-positive issue
from FINDINGS §10.1 is more likely in the recommendation text generated by
`recommendations_for_hotspots`. This is scoped as: fix if it's recommendation text (in scope);
file a bug issue if it requires detection logic changes (out of scope). AC-19 captures this
conditionality explicitly.

## Open Questions for Human Review

1. **Permission-friction investigation**: During design or implementation, someone needs to read
   `recommendations_for_hotspots` in `unimatrix-observe/src/report.rs` fully and determine whether
   any recommendation template generates "add to allowlist" or "permission prompts" language for
   `compile_cycles` or `permission_retries` findings. The finding claim text is correct; the risk
   is in the recommendation template. If the fix is recommendation text: one-line change. If it
   requires detection-context flags: file a separate bug issue.

2. **`category_gaps` field retirement strategy**: The field is currently rendered. It should be
   suppressed from rendered output (formatter change). The question is whether to also add
   `#[serde(skip_serializing_if)]` on the field itself (deprecating it in JSON output too) or just
   stop rendering it in markdown. External JSON consumers are unknown. Recommendation: suppress in
   markdown only for col-026; deprecate the field in a follow-on.

3. **Phase Timeline "absent" case**: When `cycle_events` is empty (pre-col-024 features), the
   section is silently omitted. Should a brief note appear? The SAMPLE-REVIEW shows a full table
   with no absent case. Recommend: silently omit (consistent with how `phase_narrative` handles
   empty events — it says "No phase transitions recorded" but only when the section fires at all).

4. **Top cross-feature entries title safety**: The `get(entry_id)` call returns full `EntryRecord`
   including user-authored title text. These titles appear in the rendered output as
   `"Top cross-feature entries"`. Confirm no PII or confidentiality concern in showing entry titles
   in the retrospective output — this is a developer-facing tool in a private project context, so
   likely fine, but worth a quick confirm.

## Risks and Concerns

- **Phase Timeline computation complexity**: Slicing observations by phase windows and computing
  per-phase metrics requires careful handling of observations that occur at phase boundaries (which
  phase gets them?). The design phase must define boundary semantics explicitly (e.g., `start <=
  ts < end` with the last phase using the cycle_stop timestamp as its end).

- **Compile storm risk**: FINDINGS sample review cited a compile storm during col-026 itself (29
  cycles, 11 bursts) driven by iterative `PhaseStats` field additions. The delivery protocol should
  note this and require the `PhaseStats` struct to be defined completely before first compilation.
  This is a delivery-time concern, not a scope concern.

- **Test count growth**: The formatter has extensive tests (the `retrospective.rs` module is ~2000
  lines). New sections (Phase Timeline, What Went Well, revised Knowledge Reuse) will require
  significant new test coverage. The design phase should plan for this — AC-17 requires all
  existing tests to pass, which means the formatter changes must be carefully backward compatible.
