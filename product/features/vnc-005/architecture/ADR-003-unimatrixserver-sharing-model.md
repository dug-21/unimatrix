## ADR-003: UnimatrixServer Sharing via Clone (no Arc wrapper)

### Context

The scope document (Constraint #3) states: "UnimatrixServer must become Clone or
Arc-wrapped — currently constructed once and moved into `serve()`; daemon mode requires
constructing a handler per session or sharing a single handler behind an Arc."

The risk assessment (SR-06) identifies this as a coordinated refactor with shutdown
decoupling. Before designing the sharing model, the existing source was reviewed.

Findings:
- `UnimatrixServer` already derives `Clone` (server.rs line 95: `#[derive(Clone)]`).
- All fields are `Arc`-wrapped: `Arc<AsyncEntryStore>`, `Arc<AsyncVectorStore>`,
  `Arc<EmbedServiceHandle>`, `Arc<AgentRegistry>`, `Arc<AuditLog>`,
  `Arc<CategoryAllowlist>`, `Arc<Store>`, `Arc<VectorIndex>`, `Arc<UsageDedup>`,
  `Arc<AdaptationService>`, `Arc<Mutex<PendingEntriesAnalysis>>`,
  `Arc<SessionRegistry>`, and the `ServiceLayer` (which itself holds Arc internals).
- The `ToolRouter<Self>` and `ServerInfo` fields are also Clone (ToolRouter is
  generated by the rmcp macro; ServerInfo derives Clone).
- Clone is already cheap: it is a reference count increment per Arc field.

Two approaches:

**Option A: `Arc<UnimatrixServer>` wrapper.** The server is held as
`Arc<UnimatrixServer>`. Each session task receives `Arc::clone(&server)`. The server
implements `rmcp::ServerHandler` via the Arc (requires an `impl ServerHandler for
Arc<UnimatrixServer>` blanket or a newtype wrapper, since rmcp's `serve` takes `self`
and requires `ServerHandler`).

This approach adds complexity: rmcp's `ServiceExt::serve` moves the handler into the
service, which means the handler type must be `Clone + ServerHandler`. `Arc<T>` is
`Clone` but `Arc<UnimatrixServer>` does not automatically implement `ServerHandler`.
A newtype or blanket impl would be needed.

**Option B: Clone directly.** Each session task receives `server.clone()`, which
clones all Arc fields cheaply. The server type already implements `ServerHandler`
directly. `serve(transport)` moves a server clone into the rmcp service, which is
exactly what rmcp expects. No newtype, no blanket impl.

### Decision

Use **Option B: Clone directly**.

`UnimatrixServer` already satisfies the requirements: `#[derive(Clone)]` and all
fields are `Arc`-wrapped. No structural change to the type is needed. The daemon
startup path constructs one `UnimatrixServer`, then in the accept loop passes
`server.clone()` to each `tokio::spawn` call.

The one coordination point: the `ServiceLayer` constructed in `main.rs` is shared with
the `UnimatrixServer` via `Arc` internals. Today, the server's `UnimatrixServer::new`
constructs its own internal `ServiceLayer`. For the `Arc::try_unwrap(store)` guarantee
to hold (ADR-005 / vnc-006 fix), the `LifecycleHandles` must hold the `ServiceLayer`
that the server uses, not a duplicate. The implementation must follow the existing
pattern in `main.rs` lines 274-284: construct `ServiceLayer` externally, then wire it
into the server (the current code sets `server.pending_entries_analysis` and
`server.session_registry` after construction, which is the same mechanism). The
`services` field on `UnimatrixServer` must be set from the external `ServiceLayer` for
shutdown to succeed.

Specifically: `UnimatrixServer::new` currently creates its own internal `ServiceLayer`.
This creates a second `Arc<Store>` clone set that `LifecycleHandles` does not track.
The existing workaround in `main.rs` is that `UnimatrixServer::new` is called with the
same `Arc<Store>` clone, so the internal `ServiceLayer` increments the reference count,
and `LifecycleHandles` holds the `services` from the external `ServiceLayer`.

For daemon mode, session clones of `UnimatrixServer` each hold an `Arc<ServiceLayer>`
clone (well, `ServiceLayer` is cloned by value, but its internals are `Arc`). These
clones hold `Arc<Store>` references. All session tasks must be joined and their handles
dropped before `graceful_shutdown` calls `Arc::try_unwrap(store)`. ADR-002 documents
the join protocol that ensures this.

### Consequences

Easier:
- Zero structural change to `UnimatrixServer`. The `#[derive(Clone)]` already exists.
- `serve(transport)` works identically for stdio and UDS sessions — same type, same
  trait impl.
- No new blanket impls or newtypes.

Harder:
- Every `UnimatrixServer` clone holds a shallow copy of `ServiceLayer`. `ServiceLayer`
  clones its internal Arc fields (cheap). The `ToolRouter<Self>` field is also cloned
  on each session spawn. This is cheap (Arc increment) but should be noted.
- The implementation must NOT construct a new `ServiceLayer` per session — the server
  must be built once and cloned. Building a new `ServiceLayer` per session would create
  independent confidence state handles and effectiveness handles that diverge from the
  background tick's handles. The single-construction guarantee must be preserved.
