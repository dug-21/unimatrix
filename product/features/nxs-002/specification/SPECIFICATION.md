# nxs-002: Vector Index -- Specification

**Feature**: nxs-002 (Nexus Phase)
**Status**: Specification
**Source**: SCOPE.md, PRODUCT-VISION.md, ASS-001 D1, nxs-001 ARCHITECTURE.md

---

## 1. Objective

Provide a synchronous Rust library (`unimatrix-vector`) that wraps hnsw_rs to enable approximate nearest neighbor search over 384-dimension text embeddings, coordinating with the unimatrix-store VECTOR_MAP table for crash-safe entry-to-vector mapping. The library supports unfiltered search, metadata-filtered search, index persistence, and graceful lifecycle management.

## 2. Functional Requirements

### FR-01: Index Creation

**FR-01.1: New Empty Index**
`VectorIndex::new(store: Arc<Store>, config: VectorConfig)` creates a new hnsw_rs index with the parameters from `VectorConfig`: dimension (384), max_nb_connection (16), ef_construction (200), max_elements (10,000 default), max_layer (16). The index starts in insert mode. Returns `Result<VectorIndex>`.

**FR-01.2: VectorConfig Defaults**
`VectorConfig::default()` returns: dimension=384, max_nb_connection=16, ef_construction=200, max_elements=10_000, max_layer=16, default_ef_search=32. All fields are configurable.

**FR-01.3: Thread Safety**
`VectorIndex` implements `Send + Sync`. It is shareable via `Arc<VectorIndex>`. Concurrent searches are non-blocking. Inserts acquire exclusive access to the hnsw_rs index.

### FR-02: Vector Insertion

**FR-02.1: Insert Single Vector**
`VectorIndex::insert(entry_id: u64, embedding: &[f32]) -> Result<()>`:
1. Validates `embedding.len() == config.dimension`. Returns `VectorError::DimensionMismatch` if not.
2. Generates a new hnsw data ID via atomic counter.
3. Inserts the embedding into hnsw_rs with the data ID.
4. Writes `(entry_id, data_id)` to VECTOR_MAP via `Store::put_vector_mapping`.
5. Updates the bidirectional IdMap.

**FR-02.2: Re-embedding**
If `insert` is called with an `entry_id` that already has a vector mapping:
1. A new data ID is generated (old data ID is NOT reused).
2. The new point is inserted into hnsw_rs.
3. VECTOR_MAP is overwritten with the new data_id.
4. IdMap `entry_to_data` is updated to the new data_id.
5. The old point remains in hnsw_rs (unreachable via IdMap's forward lookup but still present in the graph).

**FR-02.3: Data ID Generation**
Data IDs are generated by an `AtomicU64` starting at 0, incremented by 1 for each insert. Data IDs are independent of entry IDs. The counter value is persisted during dump and restored during load.

### FR-03: Unfiltered Search

**FR-03.1: Basic Search**
`VectorIndex::search(query: &[f32], top_k: usize, ef_search: usize) -> Result<Vec<SearchResult>>`:
1. Validates `query.len() == config.dimension`.
2. Calls hnsw_rs `search_filter` with `filter: None`.
3. Maps each `Neighbour.d_id` to `entry_id` via IdMap reverse lookup.
4. Computes `similarity = 1.0 - Neighbour.distance`.
5. Returns results sorted by similarity descending.

**FR-03.2: Empty Index**
Searching an empty index (no vectors inserted) returns `Ok(Vec::new())` -- empty result set, no error.

**FR-03.3: Fewer Results Than top_k**
If the index has fewer than `top_k` vectors, returns all available vectors. If a filter restricts results below `top_k`, returns only matching vectors. The result count may be less than `top_k`.

**FR-03.4: ef_search Parameter**
`ef_search` controls the search beam width. Higher values produce more accurate results but slower search. Must be >= `top_k`. If `ef_search < top_k`, the method uses `top_k` as the effective `ef_search`.

### FR-04: Filtered Search

**FR-04.1: Metadata-Filtered Search**
`VectorIndex::search_filtered(query: &[f32], top_k: usize, ef_search: usize, allowed_entry_ids: &[u64]) -> Result<Vec<SearchResult>>`:
1. Translates `allowed_entry_ids` to hnsw data IDs via IdMap forward lookup.
2. Sorts the data IDs to produce a `Vec<usize>` (sorted allow-list for FilterT).
3. Calls hnsw_rs `search_filter` with the sorted allow-list as the filter.
4. Maps results back to entry IDs via IdMap reverse lookup.
5. Returns results sorted by similarity descending.

**FR-04.2: Unknown Entry IDs**
Entry IDs in `allowed_entry_ids` that do not have vector mappings are silently skipped. No error is returned.

**FR-04.3: Empty Allow-List**
If `allowed_entry_ids` is empty or none of the IDs have vector mappings, returns `Ok(Vec::new())`.

### FR-05: Index Persistence

**FR-05.1: Dump**
`VectorIndex::dump(dir: &Path) -> Result<()>`:
1. Calls `hnsw.file_dump(dir, "unimatrix")` to produce `.hnsw.graph` and `.hnsw.data` files.
2. Records the actual basename returned by hnsw_rs (may differ from "unimatrix" if files exist and are mmap'd).
3. Writes a metadata file (`unimatrix-vector.meta`) containing: basename, point_count, dimension, next_data_id.

**FR-05.2: Load**
`VectorIndex::load(store: Arc<Store>, config: VectorConfig, dir: &Path) -> Result<VectorIndex>`:
1. Reads the metadata file to recover basename and next_data_id.
2. Loads the hnsw_rs index from the graph + data files using `HnswIo::load_hnsw_with_dist(DistDot)`.
3. Iterates VECTOR_MAP entries from redb to rebuild the bidirectional IdMap.
4. Restores `next_data_id` from the persisted counter.
5. Returns a `VectorIndex` ready for search operations.

**FR-05.3: Missing Index Files**
If the index directory or index files do not exist, `load` returns `VectorError::Persistence` with a descriptive message. The caller should create a new empty index and trigger re-embedding.

### FR-06: Index Inspection

**FR-06.1: Point Count**
`VectorIndex::point_count() -> usize` returns the number of vectors in the hnsw_rs index (includes stale re-embedded points).

**FR-06.2: Contains**
`VectorIndex::contains(entry_id: u64) -> bool` returns whether the given entry has a current vector mapping in the IdMap.

**FR-06.3: Stale Count**
`VectorIndex::stale_count() -> usize` returns `point_count() - active_mapping_count()` -- the number of hnsw_rs points that are no longer the current mapping for any entry (from re-embedding).

### FR-07: SearchResult Type

```rust
#[derive(Debug, Clone, Copy, PartialEq)]
pub struct SearchResult {
    pub entry_id: u64,
    pub similarity: f32,
}
```

`similarity` is computed as `1.0 - distance` where distance is from DistDot. For normalized vectors:
- 1.0 = identical vectors
- 0.0 = orthogonal vectors
- Negative values are possible for opposite-direction vectors

Results are always sorted by similarity descending (most similar first).

## 3. Non-Functional Requirements

### NFR-01: Performance

- Insert latency: < 1ms per vector at index sizes up to 10K (sub-millisecond insert is expected from hnsw_rs benchmarks).
- Search latency: < 10ms for top-10 search at index sizes up to 10K at ef_search=32.
- These targets are based on hnsw_rs published benchmarks (62K queries/sec at 784d) scaled to our lower dimension (384d).
- A benchmark test validates `set_searching_mode` overhead at 100, 1K, and 10K index sizes.

### NFR-02: Memory

- At 384 dimensions, M=16: ~1.8 MB per 1K entries, ~18 MB per 10K entries, ~183 MB per 100K entries.
- IdMap overhead: ~32 bytes per entry (two HashMaps). Negligible compared to hnsw_rs.
- Total memory budget: vector index + IdMap + VECTOR_MAP in redb cache. At 10K entries: ~18 MB (acceptable for local-first).

### NFR-03: Safety

- `#![forbid(unsafe_code)]` at crate level. No unsafe blocks.
- All public functions return `Result<T, VectorError>`. No panics.

### NFR-04: Compatibility

- Rust edition 2024, MSRV 1.89 (workspace setting).
- hnsw_rs v0.3.3, anndists v0.1.4.
- unimatrix-store as path dependency (same workspace).

## 5. Domain Models

### Key Entities

| Entity | Definition |
|--------|-----------|
| **VectorIndex** | The hnsw_rs wrapper that manages the ANN index, VECTOR_MAP coordination, and ID mapping. One per Store instance. |
| **SearchResult** | A search hit containing `entry_id` and `similarity` score. Returned by `search` and `search_filtered`. |
| **VectorConfig** | Configuration for the index: dimension, M, ef_construction, max_elements, max_layer, default_ef_search. |
| **IdMap** | Internal bidirectional map between entry IDs (u64) and hnsw data IDs (u64). Rebuilt from VECTOR_MAP on load. |
| **EntryIdFilter** | Internal FilterT implementation: sorted `Vec<usize>` of allowed hnsw data IDs for filtered search. |
| **Embedding** | A `Vec<f32>` or `&[f32]` of exactly 384 dimensions representing the semantic content of an entry. |
| **Data ID** | A monotonically increasing `u64` assigned by `VectorIndex` to each inserted vector. Maps 1:1 to an hnsw_rs internal point. Independent of entry IDs. |
| **VECTOR_MAP** | The redb table (defined in nxs-001) storing `entry_id -> data_id` mappings. Source of truth for crash recovery. |

### Relationships

```
EntryRecord (nxs-001)
    |
    | 0..1 (zero if not yet embedded)
    v
VECTOR_MAP entry: entry_id -> data_id
    |
    | 1
    v
hnsw_rs point: (embedding, data_id)
    |
    | N..1 (multiple data_ids can map to same entry_id via re-embedding)
    v
IdMap: data_id <-> entry_id (bidirectional)
```

### Ubiquitous Language

- **Embedding**: A 384-dimension float vector representing the semantic content of an entry's title + content.
- **Similarity**: `1.0 - distance`. Range typically [0.0, 1.0] for normalized vectors. Higher = more similar.
- **Distance**: Raw DistDot output. `1 - dot(a, b)` for normalized vectors. Lower = more similar.
- **Data ID**: The numeric identifier assigned by VectorIndex to each hnsw_rs point. Not the same as entry ID.
- **Allow-list**: A set of entry IDs that restricts filtered search results. Built by the caller from metadata queries.
- **Stale point**: An hnsw_rs point whose data_id is no longer the current mapping for its entry (from re-embedding). Benign but wastes memory.
- **Mode transition**: Switching hnsw_rs between insert mode and search mode via `set_searching_mode`. Requires exclusive access.

## 6. User Workflows

### Workflow 1: First-Time Index Setup

1. Caller creates `Store::open(path)` (nxs-001).
2. Caller creates `VectorIndex::new(Arc::new(store), VectorConfig::default())`.
3. Index is empty, ready for inserts.

### Workflow 2: Embedding and Inserting a New Entry

1. Caller inserts an entry via `Store::insert(new_entry)` -- returns `entry_id`.
2. Caller generates embedding via nxs-003: `embedding = embed(entry.title + entry.content)`.
3. Caller calls `vector_index.insert(entry_id, &embedding)`.
4. VectorIndex validates dimension, inserts into hnsw_rs, writes VECTOR_MAP, updates IdMap.

### Workflow 3: Semantic Search (Unfiltered)

1. Caller generates query embedding: `query_embedding = embed(query_text)`.
2. Caller calls `vector_index.search(&query_embedding, top_k=10, ef_search=32)`.
3. VectorIndex returns `Vec<SearchResult>` sorted by similarity descending.
4. Caller hydrates results: `store.get(result.entry_id)` for each result.

### Workflow 4: Metadata-Filtered Semantic Search

1. Caller queries store for allowed entries: `store.query_by_status(Active)` -> entry IDs.
2. Optionally further filter by topic/category/tags.
3. Caller calls `vector_index.search_filtered(&query_emb, 10, 32, &allowed_ids)`.
4. VectorIndex translates entry IDs to data IDs, builds FilterT, searches.
5. Returns only entries in the allow-list.

### Workflow 5: Persistence (Clean Shutdown)

1. Caller calls `vector_index.dump(&index_dir)`.
2. VectorIndex dumps hnsw_rs files + writes metadata.
3. Caller calls `store.compact()`.
4. Both database and index are safely persisted.

### Workflow 6: Persistence (Startup)

1. Caller calls `Store::open(db_path)`.
2. Caller calls `VectorIndex::load(store, config, index_dir)`.
3. VectorIndex loads hnsw_rs from files, rebuilds IdMap from VECTOR_MAP.
4. Index is ready for search.

### Workflow 7: Crash Recovery

1. Store opens normally (redb is crash-safe).
2. `VectorIndex::load` fails (hnsw_rs files missing or corrupt).
3. Caller creates new empty index: `VectorIndex::new(store, config)`.
4. Caller triggers re-embedding for all entries that have VECTOR_MAP mappings.
5. Re-embedding restores the index.

## 7. Constraints

- Rust edition 2024, MSRV 1.89 (workspace setting).
- `#![forbid(unsafe_code)]` at crate level.
- Synchronous API only (no async runtime dependency).
- Fixed dimension: 384 (all-MiniLM-L6-v2). Enforced on every insert and search.
- Fixed distance metric: DistDot. Set at index creation.
- VECTOR_MAP is source of truth for crash recovery.
- No deletion from hnsw_rs index (library limitation).
- `set_searching_mode` requires exclusive access (RwLock write lock).

## 9. NOT in Scope

- Embedding generation (nxs-003).
- MCP tool exposure (vnc-001, vnc-002).
- Async API (consumer responsibility).
- Near-duplicate detection logic (vnc-002 uses search, but detection logic is not in this crate).
- Confidence scoring (crt-002).
- Index rebuild/garbage collection tooling.
- Multi-index support (one index per store).
- Batch insert API (caller loops over individual inserts; parallel_insert optimization is a future enhancement).
- Custom distance metrics (DistDot is fixed).

## 10. Open Questions (for Implementation)

**OQ-1: VECTOR_MAP Iteration**
The `load` path needs to iterate all VECTOR_MAP entries to rebuild IdMap. unimatrix-store does not currently expose this method. Recommendation: add `Store::iter_vector_mappings() -> Result<Vec<(u64, u64)>>` to unimatrix-store during nxs-002 implementation.

**OQ-2: Hnsw Lifetime Parameter**
`Hnsw<'static, f32, DistDot>` requires the `'static` lifetime. Verify during implementation that owned `Vec<f32>` data passed via `insert_slice` satisfies this requirement.

**OQ-3: Metadata File Format**
The simple line-based format for `unimatrix-vector.meta` is sufficient for now. If more metadata is needed in the future, consider JSON or TOML. For nxs-002, keep it simple.
