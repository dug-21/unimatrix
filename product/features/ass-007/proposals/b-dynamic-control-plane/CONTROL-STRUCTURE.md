# Proposal B: Dynamic Control Plane -- Control Structure

## What Stays in `.claude/` (and Why)

Almost nothing is authored here. `.claude/` is an output directory.

| File | Source | Why it exists on disk |
|------|--------|----------------------|
| `CLAUDE.md` (unimatrix block) | DB type=`constitutional` | Claude Code reads CLAUDE.md at session start. Cannot be replaced. |
| `agents/ndp/*.md` | DB type=`role` | Claude Code reads agent defs when spawning subagents. |
| `protocols/*.md` | DB type=`protocol` | Agents read protocol files by path reference. |
| `protocols/agent-routing.md` | DB type=`routing` | Scrum-master reads this for team composition. |
| `rules/*.md` | DB type=`rule` | Claude Code loads contextual rules by file-glob match. |
| `skills/*/SKILL.md` | DB type=`skill` | Skills are file-based by Claude Code design. |

Every file above carries a header: `<!-- Generated by unimatrix export. Do not edit. Source: entry:{id} v{version} -->`.

Human edits to generated files are detected by `unimatrix status` (checksum mismatch) and can be imported back via `unimatrix import --file .claude/protocols/implementation-protocol.md`.

## What Lives in Unimatrix Entries

Everything in the table above, plus:

- **Domain knowledge** (conventions, patterns, ADRs) -- type=`knowledge`, never exported
- **Retrospective entries** -- type=`retrospective`, linked to feature/protocol
- **Correction chains** -- any entry type can be corrected; chain is `supersedes`/`superseded_by`
- **Template packs** -- type=`template`, used by `unimatrix seed`

## How `unimatrix export` Works

```
unimatrix export [--dry-run] [--filter type=protocol] [--out .claude/]

1. Query all entries where type IN (constitutional, role, protocol, routing, rule, skill)
   AND status = active (follow correction chains to current version)
2. For each entry, apply its RENDERER:
   - constitutional -> append block in CLAUDE.md (between markers)
   - role -> agents/ndp/{entry.topic}.md (full markdown from content field)
   - protocol -> protocols/{entry.topic}.md
   - routing -> protocols/agent-routing.md (assembled from all routing entries)
   - rule -> rules/{entry.topic}.md
   - skill -> skills/{entry.topic}/SKILL.md
3. Write header comment with entry ID + version
4. Compute SHA-256 checksum, store in .unimatrix/checksums.json
5. Report: "Exported 23 entries -> 19 files (4 routing entries merged into 1 file)"
```

`--dry-run` shows what would change without writing. `unimatrix diff` compares current `.claude/` against what export would produce.

## How Humans Manage the Control Plane

**Primary interface: CLI.** Files are for reading, not editing.

```bash
# Browse
unimatrix list --type protocol              # list all protocols
unimatrix show implementation-protocol      # render entry to terminal
unimatrix log implementation-protocol       # version history with diffs
unimatrix search "wave ordering"            # semantic search across control plane

# Modify
unimatrix edit implementation-protocol      # opens $EDITOR with entry content, saves on close
unimatrix correct implementation-protocol --reason "retro findings" # create correction
unimatrix revert implementation-protocol --to v3  # revert to specific version

# Lifecycle
unimatrix export                            # regenerate .claude/ files
unimatrix status                            # show pending corrections, checksum mismatches
unimatrix review                            # approve/reject queued auto-corrections
unimatrix diff                              # show what export would change
```

## New Project Setup

Three paths:

**Path A: From existing `.claude/` directory (migration)**
```bash
unimatrix seed --from .claude/
# Parses every .md file, infers type from path:
#   agents/ndp/*.md -> type=role
#   protocols/*.md -> type=protocol
#   rules/*.md -> type=rule
#   skills/*/SKILL.md -> type=skill
# Creates entries, embeds content, generates checksums
# Result: .claude/ files now have "Generated by" headers
```

**Path B: From template pack (new project)**
```bash
unimatrix seed --template rust-workspace
# Template packs are published entry collections
# Contains: base protocols, common roles, standard rules
# User customizes via `unimatrix edit` after seeding
```

**Path C: Empty start (minimal)**
```bash
unimatrix init
# Creates DB, adds 3-5 constitutional entries from CLAUDE.md scan
# No protocols, no roles -- user builds up incrementally
# Agents use server instructions alone (Tier 1)
```

## Process Evolution (Automated Correction Chains)

```
Feature NXS-003 retrospective:
  Agent stores: context_store(
    topic: "implementation-protocol",
    category: "retrospective",
    content: "Wave 2 agents lacked architecture context. 3 of 4 agents
              drifted because they didn't read ARCHITECTURE.md.",
    tags: ["protocol-friction", "nxs-003"]
  )

Feature NXS-004 retrospective:
  Similar entry stored, same topic, similar content.

Feature NXS-005 retrospective:
  Third similar entry. System detects cluster:
    3 retrospective entries for topic "implementation-protocol"
    similarity > 0.85 between all three
    all cite "architecture context" as the friction point

  System generates correction proposal:
    context_correct(
      original_id: <implementation-protocol entry>,
      content: <updated protocol with mandatory ARCHITECTURE.md in every agent prompt>,
      reason: "3 retrospectives (NXS-003, NXS-004, NXS-005) cite missing architecture
               context in wave 2 agents. Adding mandatory architecture file to spawn template."
    )

  If review gate enabled: queued for `unimatrix review`
  If auto mode: applied immediately

  Next `unimatrix export` regenerates .claude/protocols/implementation-protocol.md
  Next swarm reads the improved protocol.
```

The threshold (3 similar retrospectives) and auto-apply policy are configurable via `unimatrix config set correction.auto_threshold 3` and `correction.require_review true|false`.
