# Proposal B: Dynamic Control Plane -- Assumptions

## Core Strategic Assumption

Unimatrix owns the full control plane. `.claude/` files are generated artifacts -- renderings of database state, not source of truth. Protocols, routing rules, role definitions, conventions, and constitutional rules all live as Unimatrix entries with types, versions, and correction chains. The learning loop operates on both knowledge AND process.

## Ownership Boundaries

| Concern | Owner | `.claude/` Role |
|---------|-------|-----------------|
| Process flow (wave ordering, gates) | Unimatrix DB, type=`protocol` | Generated by `unimatrix export` |
| Role definitions (scope, self-checks) | Unimatrix DB, type=`role` | Generated as `agents/ndp/*.md` |
| Routing rules (task -> team shape) | Unimatrix DB, type=`routing` | Generated as `protocols/agent-routing.md` |
| Constitutional rules | Unimatrix DB, type=`constitutional` | Generated into `CLAUDE.md` append block |
| Contextual rules | Unimatrix DB, type=`rule` | Generated as `rules/*.md` |
| Skills | Unimatrix DB, type=`skill` | Generated as `skills/*/SKILL.md` |
| Domain knowledge | Unimatrix DB, type=`knowledge` | NOT exported (runtime only) |
| Corrections | Unimatrix DB, correction chains | Applied to source entries, re-exported |

## The Learning Loop

```
Agents work -> discover patterns -> context_store() -> knowledge accumulates
                                                            |
Agents work -> hit friction -> retrospective entry stored   |
                                    |                       |
System detects: 3 retros cite same protocol weakness -------+
                                    |
context_correct(protocol_entry, improved_version, reason="3 retrospectives cited...")
                                    |
unimatrix export -> .claude/ files regenerated with corrected protocol
                                    |
Next swarm reads updated .claude/ files -> improved process
```

The loop is **closed**. Process corrections flow from retrospective data through correction chains into regenerated `.claude/` files without human intervention. Humans approve via `unimatrix review` (opt-in gate) or trust the automation.

## What's Explicitly OUT of Scope

- **Runtime workflow state.** Wave progress, agent liveness, task queues -- not Unimatrix's concern.
- **Code generation.** Unimatrix generates `.claude/` control files, never application code.
- **Cross-project sync in v0.1.** Single-project control plane first.
- **UI in v0.1.** CLI is the human interface. UI is v0.3+.

## Risks (Honest Assessment)

| Risk | Severity | Mitigation |
|------|----------|------------|
| Human loses visibility into control plane | High | `unimatrix diff` shows pending changes before export. `unimatrix show` renders any entry. |
| Automated protocol correction introduces regressions | High | Review gate: `unimatrix review` queues corrections for human approval. Default=auto in CI, gated in production. |
| Bootstrapping complexity | Medium | `unimatrix seed` imports from existing `.claude/` directory OR from a template pack. Day-one is `unimatrix seed --from .claude/`. |
| CLI must be excellent or adoption fails | High | CLI is a first-class deliverable, not an afterthought. Without good CLI, humans cannot manage what they cannot see as files. |
| Database corruption loses entire control plane | Medium | `unimatrix export` is also the backup mechanism -- `.claude/` files ARE the disaster recovery format. `unimatrix seed --from .claude/` restores. |
| Over-automation: system "learns" bad patterns | Medium | Correction chains are auditable. `unimatrix log --type protocol` shows full version history. Human can `unimatrix revert`. |
