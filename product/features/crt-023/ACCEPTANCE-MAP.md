# crt-023 Acceptance Criteria Map

| AC-ID | Description | Verification Method | Verification Detail | Status |
|-------|-------------|--------------------|--------------------|--------|
| AC-01 | `NliProvider` in `unimatrix-embed` implements `CrossEncoderProvider` (trait with `score_pair`, `score_batch`, `name`). Given any valid `(query, passage)` pair, `score_pair` returns `NliScores` where `entailment + neutral + contradiction` is within `1e-4` of `1.0`. `NliProvider` is `Send + Sync`. | test | Unit test calling `score_pair` on a fixture pair and asserting sum constraint; compile-time `Send + Sync` bound | PENDING |
| AC-02 | `NliProvider` holds a `Mutex<Session>` for ONNX inference and a `Tokenizer` outside the mutex. A concurrent test with two threads calling `score_pair` simultaneously completes without deadlock or data race. | test | Compile-time trait check (`impl Send + Sync`) + concurrent unit test with two threads on the same `NliProvider` | PENDING |
| AC-03 | Input truncation is enforced inside `NliProvider` before tokenization. Inputs of 10,000 characters on either side do not panic, return an error, or produce OOM; they return valid `NliScores`. | test | Unit test with 10,000-char query and 10,000-char passage; assert no panic and `NliScores` sum within 1e-4 of 1.0 | PENDING |
| AC-04 | `NliModel` enum in `unimatrix-embed/src/model.rs` includes at minimum `NliMiniLM2L6H768` with `model_id = "cross-encoder/nli-MiniLM2-L6-H768"`. Methods `onnx_repo_path()`, `onnx_filename()`, and `cache_subdir()` return non-empty strings following the same naming conventions as `EmbeddingModel` variants. | test | Unit test asserting each method return value is non-empty and `model_id` equals the expected string | PENDING |
| AC-05 | `NliServiceHandle` in `unimatrix-server/src/infra/nli_handle.rs` implements the `Loading → Ready | Failed → Retrying` state machine. `get_provider()` returns `Err(ServerError::NliNotReady)` when loading and `Err(ServerError::NliFailed)` when retries are exhausted. A server instance with a missing NLI model file starts successfully and serves all MCP requests. | test | Integration test with missing model file: assert server starts, `context_search` returns results (cosine fallback) | PENDING |
| AC-06 | SHA-256 hash verification: when `nli_model_sha256` is present in config and does not match the model file on disk, `NliServiceHandle` transitions to `Failed`, emits a `tracing::error!` log line containing the substrings "security" and "hash mismatch", and the server continues operating. All MCP tools remain functional. | test | Test with valid model file and wrong 64-char hash; assert log contains "security" and "hash mismatch"; assert `context_search` returns results | PENDING |
| AC-07 | `[inference]` config section contains all ten NLI fields (`nli_enabled`, `nli_model_name`, `nli_model_path`, `nli_model_sha256`, `nli_top_k`, `nli_post_store_k`, `nli_entailment_threshold`, `nli_contradiction_threshold`, `max_contradicts_per_tick`, `nli_auto_quarantine_threshold`) with the specified defaults and types. A config file omitting all NLI fields deserializes successfully with defaults applied. | test | Unit test deserializing empty `[inference]` section; assert each of the ten fields has its specified default value | PENDING |
| AC-08 | Search re-ranking: when NLI is `Ready` and `nli_enabled = true`, `SearchService::search` expands the HNSW candidate pool to `nli_top_k`, scores each `(query, candidate_text)` pair via the rayon pool using `spawn_with_timeout(MCP_HANDLER_TIMEOUT)`, sorts by NLI entailment score descending, and returns the top-K results. When NLI is not ready or `nli_enabled = false`, the existing `rerank_score` path is used unchanged. | test | Integration test with NLI mock returning known scores; assert result ordering matches expected entailment sort | PENDING |
| AC-09 | Eval gate: `unimatrix eval run` results comparing baseline profile (NLI disabled) and MiniLM2 candidate profile (NLI enabled) on a snapshot with real query history must show: (a) aggregate P@K or MRR for the candidate >= baseline across all scenarios, AND (b) zero-regression section in `unimatrix eval report` is empty or all regressions individually approved by the human reviewer. This gate must be satisfied before the feature is marked deliverable. | manual | Human review of `unimatrix eval report` output; eval artifacts stored in feature delivery report | PENDING |
| AC-10 | Post-store detection: after a successful `context_store`, a fire-and-forget tokio task executes NLI detection on the top-`nli_post_store_k` HNSW neighbors. Pairs above `nli_entailment_threshold` produce a `Supports` edge; pairs above `nli_contradiction_threshold` produce a `Contradicts` edge. Both use `write_pool_server()` directly with `source='nli'`, `bootstrap_only=0`. Total new edges per call capped at `max_contradicts_per_tick`. | test | Integration test storing entry with known contradictory neighbor; assert `Contradicts` edge written with `source='nli'`, `bootstrap_only=0` | PENDING |
| AC-11 | NLI metadata in edges: every edge written by the NLI pipeline (post-store and bootstrap promotion) includes a `metadata` JSON string with at minimum the keys `nli_entailment` (f32) and `nli_contradiction` (f32). | test | Integration test asserting `metadata` column content for a written edge parses as valid JSON with both required keys | PENDING |
| AC-12 | Bootstrap edge promotion: on the first background tick after startup (when NLI is ready), all `GRAPH_EDGES` rows with `bootstrap_only=1` are processed. Confirmed rows are replaced with `source='nli'`, `bootstrap_only=0`; refuted rows are deleted. On a database with zero `bootstrap_only=1` rows, the task completes without error and sets the completion marker. Running the task again (restart) is a no-op due to durable marker in `COUNTERS`. | test | (a) Unit test with zero-row case asserting `bootstrap_nli_promotion_done` counter set to 1; (b) integration test with synthetic `bootstrap_only=1` rows asserting promotion and deletion | PENDING |
| AC-13 | Circuit breaker: total edges written per `context_store` call is capped at `max_contradicts_per_tick`. Excess pairs that would exceed the cap are dropped with a `tracing::debug!` log and not written to `GRAPH_EDGES`. Cap applies to all edge types combined (Supports + Contradicts). | test | Integration test with `max_contradicts_per_tick=2` and 5 neighbor pairs all above both thresholds; assert exactly 2 edges written total across both edge types | PENDING |
| AC-14 | Graceful degradation: with `nli_enabled=false`, or with no NLI model file present, or with hash mismatch, the server starts and handles all MCP requests using cosine-similarity fallback. No error returned to callers. A single `tracing::warn!` at startup notes NLI unavailability. | test | Startup test with each of three degradation conditions; assert: (a) startup succeeds, (b) `context_search` returns results, (c) exactly one warn-level log mentioning NLI | PENDING |
| AC-15 | Panic containment: if an NLI inference call panics on the rayon thread, `rayon_pool.spawn_with_timeout` returns `Err(RayonError::Cancelled)`. The search service maps this to cosine fallback; the MCP handler thread is not affected. | test | Test injecting a panic via mock `CrossEncoderProvider`; assert MCP handler returns successful response with fallback results | PENDING |
| AC-16 | `unimatrix model-download --nli` downloads the configured NLI model to the cache directory and prints the SHA-256 hash of the downloaded file in a format suitable for copy-paste into `nli_model_sha256` in `config.toml`. | shell | CLI test (or manual smoke test in delivery report) asserting download succeeds and SHA-256 hash is printed to stdout | PENDING |
| AC-17 | Config validation: all ten NLI config fields are validated at startup. The following abort with a structured error naming the offending field: `nli_top_k` outside `[1, 100]`; `nli_post_store_k` outside `[1, 100]`; `nli_entailment_threshold` outside `(0.0, 1.0)`; `nli_contradiction_threshold` outside `(0.0, 1.0)`; `max_contradicts_per_tick` outside `[1, 100]`; `nli_model_name` set to unrecognized variant; `nli_model_sha256` not 64 hex chars when set; `nli_auto_quarantine_threshold` outside `(0.0, 1.0)`. Cross-field: `nli_auto_quarantine_threshold <= nli_contradiction_threshold` aborts naming both fields. | test | Unit tests for each out-of-range case and cross-field violation; assert startup error message contains the offending field name(s) | PENDING |
| AC-18 | `EvalServiceLayer::from_profile()` fills the W1-4 stub: profiles with `nli_enabled=true` and a resolvable model construct `NliServiceHandle` and wire it into `SearchService`. Profiles with `nli_enabled=false` or absent NLI fields use baseline cosine path. An `eval run` with both profiles produces two result sets enabling A/B comparison. | test | Integration test running `eval run` with baseline and candidate TOML profiles against fixture snapshot; assert two result files produced | PENDING |
| AC-19 | `nli_post_store_k` (default 10) is distinct from `nli_top_k` (default 20) in config and in code. `StoreService` uses `nli_post_store_k` for neighbor retrieval; `SearchService` uses `nli_top_k` for candidate expansion. Setting one does not affect the other. | test | Unit test asserting each service reads the correct config field; set different values for each and verify each service uses its own field | PENDING |
| AC-20 | NLI entailment score replaces `rerank_score` entirely in the search re-ranking sort step when NLI is active. `rerank_score(similarity, confidence, cw)` is not called on the NLI-active sort path. | AC-21 | Model selection via config string is supported. Setting `nli_model_name = "minilm2"` selects `NliMiniLM2L6H768`; setting `nli_model_name = "deberta"` selects `NliDebertaV3Small` (if available). An unrecognized model name fails startup validation with a structured error. | test | Unit test asserting `NliModel::from_config_name` resolves each valid string identifier; assert `"gpt4"` returns `None` and triggers startup abort | PENDING |
| AC-22 | The eval gate waiver condition (D-01) is applied correctly: when `unimatrix eval scenarios` returns zero rows for the available snapshot, the gate is documented as waived in the delivery report. | AC-23 | `max_contradicts_per_tick` semantics are per-`context_store` call, not globally per background tick. The config field name is retained for compatibility. | AC-24 | Bootstrap promotion idempotency is achieved via a durable marker in the `COUNTERS` table (key `bootstrap_nli_promotion_done = 1`), not by row absence as a guard. | AC-25 | NLI-origin auto-quarantine uses a higher confidence threshold.
