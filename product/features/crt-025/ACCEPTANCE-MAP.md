# crt-025 Acceptance Criteria Map

GH #330 | Schema v14 → v15

| AC-ID | Description | Verification Method | Verification Detail | Status |
|-------|-------------|---------------------|---------------------|--------|
| AC-01 | `CycleParams` has `phase`, `outcome`, `next_phase` fields and no `keywords` field. Deserialization of JSON containing `keywords` succeeds and silently discards it (no `deny_unknown_fields`). | test | `serde_json::from_str::<CycleParams>(r#"{"type":"start","topic":"crt-025","keywords":["k"]}"#)` succeeds; `CycleParams` has no `keywords` accessor | PASS |
| AC-02 | `validate_cycle_params` accepts `"phase-end"` as a valid type returning `CycleType::PhaseEnd`. `"start"` and `"stop"` remain valid. All other values are rejected with a descriptive error naming all three valid values. | test | Unit tests: `"phase-end"` → `Ok(PhaseEnd)`; `"pause"` → `Err(...)` containing all valid names; `""` → `Err(...)` | PASS |
| AC-03 | A `phase` value containing a space character is rejected. A value exceeding 64 characters is rejected. Normalization to lowercase is applied before validation. `"Scope"` normalizes to `"scope"`. | test | Unit tests: `"scope review"` → `Err(...)`; `"a".repeat(65)` → `Err(...)`; `"Scope"` → `Ok` with `phase = Some("scope")`; `"IMPLEMENTATION"` → `Some("implementation")` | PASS |
| AC-04 | `phase` on any event type, if provided and valid, is stored in `CYCLE_EVENTS.phase`. If absent on `"start"`, stored as `NULL`. | test | Integration test: `context_cycle(type="start", topic=...)` without `phase`; query `SELECT phase FROM cycle_events WHERE cycle_id = ?`; assert `IS NULL` | PASS |
| AC-05 | `next_phase` on `"start"` immediately sets `SessionState.current_phase = Some(next_phase)`. A subsequent `context_store` in the same session receives the non-NULL phase for tagging. | test | Integration test: `start` with `next_phase="scope"`, then `context_store`; assert `feature_entries.phase = "scope"` | PASS |
| AC-06 | `next_phase` on `"phase-end"` updates `SessionState.current_phase`. If `next_phase` is absent on `"phase-end"`, `current_phase` is left unchanged. | test | Unit test on `SessionState` mutation logic for both cases | PASS |
| AC-07 | `"stop"` event clears `SessionState.current_phase` to `None`. A subsequent `context_store` in the same session receives `phase = NULL`. | test | Integration test: `stop` event, then `context_store`; assert `feature_entries.phase IS NULL` | PASS |
| AC-08 | Each `context_cycle` call produces exactly one INSERT to `CYCLE_EVENTS` with a monotonically increasing `seq` value scoped to the `cycle_id`. Three sequential calls for the same topic produce `seq = 0, 1, 2`. | test | Integration test: 3 sequential `context_cycle` calls for same `cycle_id`; assert rows exist with `seq` values 0, 1, 2 in `ORDER BY timestamp ASC, seq ASC` | PASS |
| AC-09 | `context_store` writes the current `SessionState.current_phase` to `feature_entries.phase` at insert time. Entries stored before any phase transition receive `phase = NULL`. | test | Integration tests: store with active phase → non-NULL `feature_entries.phase`; store before any phase signal → `feature_entries.phase IS NULL` | PASS |
| AC-10 | Schema version advances from 14 to 15. Migration is idempotent (running twice produces no error, schema unchanged). `CYCLE_EVENTS` table and `feature_entries.phase` column exist after migration. | test | Migration integration test: run `run_main_migrations` on v14 DB; assert schema_version=15, `cycle_events` table present, `feature_entries.phase` column present; run again, assert no error | PASS |
| AC-11 | `create_tables_if_needed` in `db.rs` creates `cycle_events` table and `feature_entries` with `phase` column on a fresh database. | test | Fresh-DB integration test: call `create_tables_if_needed`; verify via `pragma_table_info` that `cycle_events` table exists and `feature_entries` has `phase` column; assert `CURRENT_SCHEMA_VERSION = 15` | PASS |
| AC-12 | `context_cycle_review` response includes a phase narrative section when `CYCLE_EVENTS` data exists: ordered phase list, rework flag for repeated phases, per-phase category counts from `FEATURE_ENTRIES`. | test | Integration test: seed `CYCLE_EVENTS` rows for a feature, call `context_cycle_review`, assert `phase_narrative` field is present and non-null in JSON response, contains expected phases and rework flags | PASS |
| AC-13 | `context_cycle_review` response is unchanged (no phase narrative section) when no `CYCLE_EVENTS` data exists for the queried feature cycle. | test | Integration test: call `context_cycle_review` for a pre-WA-1 feature cycle (no `CYCLE_EVENTS` rows); assert JSON response does not contain `phase_narrative` key (field omitted via `skip_serializing_if = "Option::is_none"`) | PASS |
| AC-14 | `context_cycle_review` cross-cycle comparison is included when 2 or more prior features have phase-tagged `feature_entries` rows. Silently omitted when fewer than 2 prior features have phase data. | test | Integration tests: 0 prior features → `cross_cycle_comparison = None`; 1 prior feature → `None`; 2 prior features → `Some(...)` with correct mean; current feature excluded from baseline (self-exclusion) | PASS |
| AC-15 | `"outcome"` is removed from `CategoryAllowlist`. `context_store` with `category = "outcome"` returns `ServerError::InvalidCategory`. `CategoryAllowlist::new()` has exactly 7 categories. | test | Unit test: `CategoryAllowlist::new()` count = 7; `al.validate("outcome")` returns `Err(...)`; integration test: `context_store(category="outcome")` returns `InvalidCategory` error | PASS |
| AC-16 | Hook path handles `"phase-end"` events by emitting `cycle_phase_end` event type. On validation failure (e.g., `phase = "scope review"`), hook logs a warning and falls through to generic observation path without hard-failing or returning error to transport. | test | Unit test: hook event with `type="phase-end"`, `phase="scope review"` (invalid); assert warning logged and no error returned to transport; assert valid `phase-end` event emits `cycle_phase_end` and updates `SessionState.current_phase` | PASS |
| AC-17 | All new database operations are tested: `CYCLE_EVENTS` insert for all three event types, `feature_entries` phase column write (both NULL and non-NULL), schema v14→v15 migration idempotency. | test | Covered by AC-08, AC-09, AC-10 — verify all three event types produce a row in `cycle_events` with correct `event_type` value (`cycle_start`, `cycle_phase_end`, `cycle_stop`) | PASS |
