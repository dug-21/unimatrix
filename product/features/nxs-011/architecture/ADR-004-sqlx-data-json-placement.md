## ADR-004: sqlx-data.json Placement

### Context

`sqlx::query!()` macros require compile-time schema access. In online mode, this means a
live database connection during `cargo build`. In offline mode (`SQLX_OFFLINE=true`), it
reads from a `sqlx-data.json` file containing the query type information derived from the
schema.

`cargo sqlx prepare --workspace` produces a single `sqlx-data.json` at the workspace root
covering all crates in the workspace. Per-crate alternatives exist (`cargo sqlx prepare`
without `--workspace`, run from within each crate directory), which produce per-crate
`.sqlx/` directories.

**Options considered:**

**Option A: Per-crate `.sqlx/` directories**

`cargo sqlx prepare` (without `--workspace`) run separately in each crate directory
produces `.sqlx/query-{hash}.json` files per crate.

Pros: crate-level isolation; regenerating one crate's queries does not affect others.
Cons: Multiple directories to manage. CI must run `cargo sqlx prepare` in each relevant
crate directory separately. Developer workflow is more complex. The workspace assumption
(SCOPE.md §Assumptions, Constraint 2) is that a single file covers all crates.

**Option B: Workspace-level `sqlx-data.json` at repository root**

`cargo sqlx prepare --workspace` run from the workspace root produces a single
`sqlx-data.json` at the repository root.

Pros: Single file, single regeneration command, single CI check. Simpler developer
workflow. Matches the recommendation in SCOPE.md.

Cons: All crates must use the same `DATABASE_URL` when preparing. For this workspace,
all sqlx-using crates operate on the same SQLite schema, so a single `DATABASE_URL`
pointing to a test database with the current schema is correct.

**Database URL for preparation:**
`cargo sqlx prepare --workspace` requires `DATABASE_URL` to be set to a database with
the current schema applied. The convention is:

```sh
DATABASE_URL="sqlite:/tmp/unimatrix-prepare.db" cargo sqlx prepare --workspace
```

Developer must first run migrations against this temporary database to bring it to schema
v12 before running prepare. This is a one-time setup; subsequent runs after schema-neutral
changes (this feature, nxs-011, makes no schema changes) only need to re-run prepare if
`sqlx::query!()` call sites change.

**CI enforcement:**
`SQLX_OFFLINE=true` set in the CI environment causes sqlx macros to read from
`sqlx-data.json` at the workspace root. If the file is absent or a query is not found in
it, the macro expands to a `compile_error!` with a message that includes the query hash and
advises running `cargo sqlx prepare`. This is human-readable (SR-03).

A `cargo sqlx check --workspace` step in CI can explicitly verify all queries against the
cached file before the build step, surfacing stale cache as a structured pre-build error
rather than a mid-compilation cryptic failure.

### Decision

Single workspace-level `sqlx-data.json` at the repository root, generated by
`cargo sqlx prepare --workspace`.

CI enforces `SQLX_OFFLINE=true` on all `cargo build` and `cargo test` steps in
`.github/workflows/release.yml`. A `cargo sqlx check --workspace` step is added before
the build step to produce a structured error on stale cache (CI-02).

Developer regeneration command:
```sh
DATABASE_URL="sqlite:/tmp/unimatrix-prepare.db" cargo sqlx prepare --workspace
```

The resulting `sqlx-data.json` is committed to the repository root alongside `Cargo.toml`
and `Cargo.lock`. It is updated in the same commit as any new or changed `sqlx::query!()`
call.

### Consequences

- AC-12 is satisfied: `sqlx-data.json` exists at workspace root; CI enforces offline mode.
- Developer workflow is a single command (`cargo sqlx prepare --workspace`) with a
  DATABASE_URL pointing to a local prepared database.
- CI-02 is implementable as a `cargo sqlx check` pre-build step.
- All workspace crates that use `sqlx::query!()` share the same schema cache. Since
  this workspace has one database schema (SQLite only, single file), this is correct.
- If a future feature adds a second database (e.g., PostgreSQL in a separate crate with a
  different URL), this decision would need revisiting. That is explicitly not in scope for
  nxs-011 (non-goal: PostgreSQL switch).
- `sqlx-data.json` must be regenerated after any `sqlx::query!()` call site is added or
  modified. Schema changes (not in scope for nxs-011) would also require regeneration.
  This is a developer discipline enforced by CI.
