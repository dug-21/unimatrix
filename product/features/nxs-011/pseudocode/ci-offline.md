# Component: sqlx-data.json + CI Offline Enforcement
## Files: `sqlx-data.json` (new, workspace root) + `.github/workflows/release.yml` (edit)

---

## Purpose

Generates and commits `sqlx-data.json` at the workspace root to enable offline compile-time
SQL validation in CI (`SQLX_OFFLINE=true`). Updates `.github/workflows/release.yml` to
enforce `SQLX_OFFLINE=true` on all build and test steps and add a `cargo sqlx check`
pre-build step. Satisfies AC-12, NF-07, CI-01, CI-02, CI-03. (ADR-004)

This component is the last in the wave structure — it requires all `sqlx::query!()`
call sites to be finalized across all crates before `cargo sqlx prepare --workspace`
is run.

---

## sqlx-data.json Generation

`sqlx-data.json` is NOT hand-authored. It is generated by the developer tool `sqlx-cli`:

```sh
# Prerequisites:
# 1. sqlx-cli installed: cargo install sqlx-cli --no-default-features --features sqlite
# 2. A local database at schema v12 (run migrations first):
DATABASE_URL="sqlite:///tmp/unimatrix-prepare.db" \
    cargo run -p unimatrix-server -- --once-migrate-and-exit 2>/dev/null || true
    # OR: manually open and migrate the DB via test helpers

# 3. Generate the cache:
DATABASE_URL="sqlite:///tmp/unimatrix-prepare.db" \
    cargo sqlx prepare --workspace

# Result: sqlx-data.json created at workspace root (next to Cargo.toml and Cargo.lock)
```

The generated file is committed to the repository:
```sh
git add sqlx-data.json
git commit -m "chore(nxs-011): generate sqlx-data.json for offline SQL validation"
```

`sqlx-data.json` must be regenerated after any `sqlx::query!()` call site is added,
modified, or removed. This is a developer discipline constraint enforced by CI-02.

## .github/workflows/release.yml Changes

### Change 1: Add SQLX_OFFLINE=true to all cargo steps

```yaml
# BEFORE (example cargo build step):
- name: Build
  run: cargo build --release

# AFTER:
- name: Build
  env:
    SQLX_OFFLINE: "true"
  run: cargo build --release
```

Apply `SQLX_OFFLINE: "true"` to ALL `run: cargo build ...` and `run: cargo test ...` steps.
This ensures the macro reads from `sqlx-data.json` instead of requiring a live database.

### Change 2: Add cargo sqlx check pre-build step

```yaml
# Add BEFORE the main build step:
- name: Verify sqlx query cache
  env:
    SQLX_OFFLINE: "true"
  run: |
    if ! command -v cargo-sqlx &> /dev/null; then
      cargo install sqlx-cli --no-default-features --features sqlite --locked
    fi
    cargo sqlx check --workspace
```

If `sqlx-data.json` is absent or stale (a new `sqlx::query!()` call is not in the cache),
`cargo sqlx check` produces an error that names the missing query hash. This is the
"human-readable error" required by CI-02 and NF-07.

### Change 3: Add rusqlite absence check (CI-03)

```yaml
# Add to CI:
- name: Verify no rusqlite in store/server crates
  run: |
    if grep -q "rusqlite" crates/unimatrix-store/Cargo.toml; then
      echo "ERROR: rusqlite found in unimatrix-store/Cargo.toml (AC-01)"
      exit 1
    fi
    if grep -q "rusqlite" crates/unimatrix-server/Cargo.toml; then
      echo "ERROR: rusqlite found in unimatrix-server/Cargo.toml (AC-13)"
      exit 1
    fi
```

### Change 4: Add spawn_blocking + lock_conn absence checks (CI)

```yaml
- name: Verify no spawn_blocking/lock_conn in store crate
  run: |
    FOUND=$(grep -rn "spawn_blocking\|Mutex::lock\|lock_conn" crates/unimatrix-store/src/ || true)
    if [ -n "$FOUND" ]; then
      echo "ERROR: blocked patterns found in unimatrix-store:"
      echo "$FOUND"
      exit 1
    fi

- name: Verify no spawn_blocking store patterns in server crate
  run: |
    FOUND=$(grep -rn "spawn_blocking.*store" crates/unimatrix-server/src/ || true)
    if [ -n "$FOUND" ]; then
      echo "ERROR: spawn_blocking store patterns found in unimatrix-server:"
      echo "$FOUND"
      exit 1
    fi

- name: Verify no AsyncEntryStore in codebase
  run: |
    FOUND=$(grep -rn "AsyncEntryStore" crates/ --include="*.rs" || true)
    if [ -n "$FOUND" ]; then
      echo "ERROR: AsyncEntryStore references remain:"
      echo "$FOUND"
      exit 1
    fi
```

### Change 5: Add format!()-in-SQL injection check (security)

```yaml
- name: Verify no SQL string interpolation in store crate
  run: |
    # Checks for format!("SELECT...{}", ...) patterns — SQL injection risk
    FOUND=$(grep -rn 'format!.*".*SELECT\|INSERT\|UPDATE\|DELETE.*{' crates/unimatrix-store/src/ || true)
    if [ -n "$FOUND" ]; then
      echo "WARNING: format! string interpolation in SQL detected — review for injection risk:"
      echo "$FOUND"
      # exit 1  # Uncomment to make this a hard failure after review
    fi
```

## Developer Workflow Documentation (README or CONTRIBUTING)

Add to `README.md` or `CONTRIBUTING.md` under "Database Development":

```markdown
## Regenerating sqlx-data.json

After adding, modifying, or removing any `sqlx::query!()` call, regenerate the query cache:

```sh
# 1. Ensure a local database at schema v12 exists:
DATABASE_URL="sqlite:///tmp/unimatrix-prepare.db" cargo test -p unimatrix-store --test migration

# 2. Regenerate the cache:
DATABASE_URL="sqlite:///tmp/unimatrix-prepare.db" cargo sqlx prepare --workspace

# 3. Commit the updated file:
git add sqlx-data.json && git commit -m "chore: regenerate sqlx-data.json"
```

CI enforces `SQLX_OFFLINE=true` and will fail if `sqlx-data.json` is absent or stale.
```

## Error Handling

- `cargo sqlx check` failure in CI: structured error with query hash; CI step fails; PR
  is blocked. Developer must regenerate `sqlx-data.json` and push an updated commit.
- `SQLX_OFFLINE=true` + missing `sqlx-data.json`: `compile_error!` macro expansion with
  human-readable message including the query hash. CI build fails at the compile step.
- rusqlite presence check failure: CI step fails with named file; developer must remove
  the dependency.

## Key Test Scenarios

1. **`test_offline_build_succeeds`** (AC-12, CI-01): Run `cargo build --offline` (or with
   `SQLX_OFFLINE=true`) from a clean checkout with `sqlx-data.json` committed; assert build
   succeeds without a live `DATABASE_URL`.

2. **`test_cargo_sqlx_check_passes`** (CI-02): Run `cargo sqlx check --workspace` with
   `SQLX_OFFLINE=true` and the committed `sqlx-data.json`; assert exit code 0.

3. **`test_no_rusqlite_in_store_server`** (AC-01, AC-13, CI-03): CI grep step returns zero
   matches for `rusqlite` in `unimatrix-store/Cargo.toml` and `unimatrix-server/Cargo.toml`.

4. **`test_sqlx_data_json_at_workspace_root`**: Assert `sqlx-data.json` exists at the
   repository root alongside `Cargo.toml`. Verifiable by CI `test -f sqlx-data.json`.

5. **`test_stale_cache_produces_human_readable_error`** (CI-02): Modify a `sqlx::query!()`
   without regenerating `sqlx-data.json`; assert `cargo sqlx check` produces an error
   message that includes the missing query hash (not a cryptic macro expansion error).
