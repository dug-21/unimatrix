# VNC-014: Server-Side MCP Session Attribution via clientInfo.name

## Problem Statement

Codex CLI (OpenAI) does not fire client-side hooks due to openai/codex#16732. Because
`cycle_events` is populated exclusively via the hook path (`hook.rs::build_cycle_event_or_fallthrough`
intercepting `PreToolUse` for `context_cycle`) and not from the MCP tool handler directly, Codex
sessions produce zero hook events. The result: the `audit_log` contains no client attribution for
Codex tool calls — every event is recorded with `agent_id = "anonymous"` unless the agent is
explicitly instructed to pass `agent_id` in each call.

The MCP `initialize` handshake gives the server `params.clientInfo.name` — a stable, client-set
string that identifies the connecting MCP client (e.g. `"codex-mcp-client"` for Codex,
`"gemini-cli-mcp-client"` for Gemini CLI). By capturing this value at initialize time, binding it
to the transport-level `Mcp-Session-Id`, and propagating it to every subsequent `audit_log` write
in that session, we get zero-friction client attribution that works for all MCP clients regardless
of whether they fire hooks.

This feature fixes the audit attribution gap for non-Claude-Code clients. It does not fix the
`cycle_events` gap (that would require server-side hook simulation, which is out of scope).

## Goals

1. At `initialize` request handling, extract `params.clientInfo.name` and bind it to the
   server-assigned rmcp session ID in a connection-level map on `UnimatrixServer`.
2. Propagate `client_type` (the raw `clientInfo.name` string) on all subsequent tool call audit
   records within that rmcp session.
3. Write `clientInfo.name` to `audit_log.agent_attribution` (dedicated non-spoofable attribution
   column, per ASS-050 schema) and as `client_type` key in the `audit_log.metadata` JSON field.
   Both columns are added in the same four-column ASS-050 migration.
4. Ship `build_context_with_external_identity()` (Seam 2, ASS-050) — the new overload that
   accepts `RequestContext` and a stub `Option<&ResolvedIdentity>`. Required by this feature to
   read the rmcp session ID; activatable by W2-3 by passing the bearer-validated identity.
5. Implement with zero behavioral regression: tools that receive no session context continue to
   work correctly — `agent_attribution` is empty string and `metadata` is `'{}'` when no rmcp
   session is active.

## Non-Goals

- **Does not fix `cycle_events` gap for Codex.** `cycle_events` is populated only via the hook
  path. Server-side attribution via `clientInfo.name` cannot populate `cycle_events` for clients
  that do not fire hooks. This gap remains and is an explicit non-goal.
- **Does not replace vnc-013 hook normalization.** The hook normalization layer (Gemini canonical
  event names) is a separate, parallel work stream. VNC-014 operates on the MCP path only.
- **Does not implement OAuth JWT identity.** The enterprise tier's JWT `sub`+`aud` superseding
  `clientInfo.name` is an enterprise-tier extension. OSS implementation only.
- **Does not add `client_type` as a tool parameter.** Attribution is server-side and transparent
  to agents — no change to any tool's parameter schema.
- **Does not implement retention, querying, or reporting on `client_type`.** The field is written;
  reads and analytics are deferred.
- **Does not add `Mcp-Session-Id` awareness to the stdio transport.** The stdio path has no HTTP
  session concept. `client_type` on stdio sessions will be captured via a connection-level
  `client_info` store on the `UnimatrixServer` (see Proposed Approach).

## Background Research

### rmcp 0.16.0 — `initialize` Hook and `clientInfo` Availability

The `ServerHandler` trait in rmcp 0.16.0 exposes an `initialize` method (defaulting to
`get_info()`) that receives the full `InitializeRequestParams`:

```rust
fn initialize(
    &self,
    request: InitializeRequestParams,
    context: RequestContext<RoleServer>,
) -> impl Future<Output = Result<InitializeResult, McpError>> + Send + '_ {
```

`InitializeRequestParams.client_info` is of type `Implementation`, which has:
- `name: String` — the client name (e.g. `"codex-mcp-client"`)
- `version: String` — the client version

This is the `params.clientInfo.name` the feature needs. It is **already available via the default
trait hook** — no rmcp fork or patch required.

### `Mcp-Session-Id` Handling

In the streamable HTTP transport (`tower.rs`), the session ID is generated by the session manager
at the first `InitializeRequest` and injected as the `Mcp-Session-Id` response header. It is also
injected into request extensions (`req.request.extensions_mut().insert(part)`) for every
subsequent request in the session. The `RequestContext` carries these extensions.

**Critical finding**: The `Mcp-Session-Id` at the rmcp transport layer is the session identifier
used by rmcp's session manager (an `Arc<str>` UUID). It is **distinct** from Unimatrix's
`session_id` field in `AuditEvent`, which is sourced from the agent-declared `session_id` tool
parameter (prefixed `mcp::`). There is no existing mechanism that connects the rmcp transport
session ID to the Unimatrix session registry or audit log automatically.

The INSERT in `unimatrix-store/src/audit.rs` (`log_audit_event`) lists 8 named columns. Adding
`metadata` requires updating both the INSERT statement and the `AuditEvent` struct. The
`read_audit_event` SELECT also needs updating to round-trip the new field.

The store's INSERT in `audit.rs` currently uses positional `?1`..`?8` binding — adding `?9` for
`metadata` is straightforward.

### `SessionRegistry` — Current State and `client_type` Storage

`SessionRegistry` (`infra/session.rs`) tracks per-session state keyed on a `session_id: String`.
Sessions are registered via `register_session(session_id, role, feature)`. The `SessionState`
struct has no `client_type` field today.

The `session_id` used as the key in `SessionRegistry` is the agent-declared `session_id` tool
parameter (prefixed with `mcp::`), not the rmcp transport-level `Mcp-Session-Id`. This is the
critical coupling point: the rmcp-level session ID must be linked to the agent-declared session ID
to enable client_type propagation through existing session_id-keyed audit paths.

### `build_context` / `audit_fire_and_forget` — Current Audit Path

In `server.rs`, `build_context` constructs the `AuditContext` from `agent_id`, `format`, and
`session_id` (agent-declared). `AuditEvent` is then built per-handler in `tools.rs`, with
`session_id: String::new()` for non-session-aware audit events, or
`ctx.audit_ctx.session_id.clone()` when a session_id is present.

There is **no existing `client_type` field** in `AuditContext`, `AuditEvent`, or `ToolContext`.
All three need extension.

### `ServerHandler::initialize` Is Not Overridden

Confirmed in `server.rs`: `UnimatrixServer` implements `impl rmcp::ServerHandler for
UnimatrixServer` with only `get_info()`. The `initialize` method is not overridden — the default
(which calls `get_info()`) is in effect. VNC-014 requires overriding `initialize` on
`UnimatrixServer` to capture `params.client_info.name`.

**Confirmed access path** (ASS-050 OQ-01, empirical): `clientInfo.name` is NOT in
`RequestContext.extensions`. It lives at:
```rust
ctx.peer.peer_info()           // Option<&InitializeRequestParams>
    .map(|ci| ci.client_info.name.as_str())
```
`peer_info()` is populated by rmcp before any tool call is dispatched. Returns `None` only
when `serve_directly()` is used (Unimatrix does not use this path).

- Gemini: `"gemini-cli-mcp-client"` (inferred, needs live capture confirmation)

ASS-050 is complete. The full four-column migration is specified and must land as a single schema

**Four new `audit_log` columns (ASS-050 Section 4)**:

## Proposed Approach

### 1. `SessionRegistry` — No Change Required

`SessionRegistry` is keyed on the agent-declared `session_id` tool parameter. `client_type`
attribution is keyed on the rmcp transport session ID (`Mcp-Session-Id`) — a different namespace.
Bridging these two is not required: `client_type` is looked up directly from `client_type_map`
(Step 2) using the rmcp session ID at `build_context` time. `SessionRegistry` is not modified.

### 2. `UnimatrixServer` — Connection-Level `client_info` Map

**OQ-01 resolved** — approach (A) is confirmed with exact access paths known.

Add `client_type_map: Arc<Mutex<HashMap<String, String>>>` to `UnimatrixServer`. This maps
rmcp-level session ID (`Mcp-Session-Id` UUID) → `clientInfo.name` string.

**rmcp session ID access path** (ASS-050 OQ-03, empirical): Available at tool call time via:
```rust
request_context.extensions
    .get::<http::request::Parts>()
    .and_then(|parts| parts.headers.get("mcp-session-id"))
    .and_then(|v| v.to_str().ok())
    .map(|s| s.to_owned())
```
rmcp injects `http::request::Parts` into `RequestContext.extensions` for every HTTP tool call
POST. The `Mcp-Session-Id` header is the server-assigned UUID, echoed by the client on every
request in the session.

**Stdio path**: Stdio has one connection per server lifetime (no multiplexing). Use the empty
string `""` as the map key for stdio — one entry, set at initialize time, valid for the
server's lifetime. The same `HashMap<String, String>` handles both cases.

### 3. Override `ServerHandler::initialize`

In `server.rs`, override `initialize`:

```rust
fn initialize(
    &self,
    request: InitializeRequestParams,
    context: RequestContext<RoleServer>,
) -> impl Future<Output = Result<InitializeResult, McpError>> + Send + '_ {
    let client_name = request.client_info.name.clone();
    if !client_name.is_empty() {
        // Key on rmcp session ID from Mcp-Session-Id header; fall back to "" for stdio
        let session_key = context.extensions
            .get::<http::request::Parts>()
            .and_then(|p| p.headers.get("mcp-session-id"))
            .and_then(|v| v.to_str().ok())
            .unwrap_or("")
            .to_owned();
        let truncated = client_name.chars().take(256).collect::<String>();
        self.client_type_map.lock().unwrap().insert(session_key, truncated);
    }
    std::future::ready(Ok(self.get_info()))
}
```

`clientInfo.name` is accessed directly from `request.client_info.name` — it is a field on the
`InitializeRequestParams` parameter, not from `context`. The `peer_info()` path confirmed in
OQ-01 applies at tool call time; at `initialize` time the params are available directly.

### 5. `build_context_with_external_identity()` — Ship Seam 2 Overload

**vnc-014 ships the `build_context_with_external_identity()` overload (Seam 2, ASS-050).** This
is a deliverable of this feature, not a pre-existing function. The existing `build_context()` has
no `RequestContext` parameter and cannot read the rmcp session ID — a new overload is required
regardless. Shipping it with the full Seam 2 signature now costs nothing and shrinks W2-3's
implementation surface to wiring the bearer middleware result through an already-existing function.

Signature:
```rust
pub(crate) async fn build_context_with_external_identity(
    &self,
    params_agent_id: &Option<String>,
    format: &Option<String>,
    session_id: &Option<String>,
    request_context: &RequestContext<RoleServer>,
    external_identity: Option<&ResolvedIdentity>, // always None in vnc-014; W2-3 wires this
) -> Result<ToolContext, rmcp::ErrorData>
```

In this function: extract the rmcp session ID from `request_context.extensions` (the
`Mcp-Session-Id` header via `http::request::Parts`), look up `client_type` in
`self.client_type_map`, attach to `ToolContext`. If `external_identity` is `Some`, bypass
`resolve_agent()` entirely (W2-3 activation path — stub `None` in vnc-014 leaves existing
`resolve_agent()` behavior unchanged).

All 12 tool handlers switch from `build_context()` to `build_context_with_external_identity()`,
passing their `ToolCallContext.request_context` and `None` for `external_identity`. The existing
`build_context()` is retained as a thin wrapper or removed — implementation agent decides based
on call-site count.

### 6. `AuditEvent` Construction — Populate All Four New Fields

Where `AuditEvent` is constructed (all sites in `tools.rs`), populate all four new fields:

```rust
AuditEvent {
    // existing fields unchanged ...
    credential_type:   "none".to_string(),       // STDIO; "static_token" once W2-2 bearer lands
    capability_used:   capability_name,           // string name of the Capability gate evaluated
    agent_attribution: ctx.client_type.clone().unwrap_or_default(), // clientInfo.name or ""
    metadata: ctx.client_type.as_deref()
        .filter(|s| !s.is_empty())
        .map(|ct| format!(r#"{{"client_type":"{}"}}"#, ct.replace('"', "\\\"")))
        .unwrap_or_else(|| "{}".to_string()),
}
```

`agent_attribution` gets `clientInfo.name` directly — it is the non-spoofable attribution
column. `metadata` also records it as `client_type` for forward-compatibility.

`capability_name` is the string name of the capability gate each tool requires (e.g. `"write"`,
`"read"`, `"admin"`). Each tool handler knows which capability it checks — pass the string
constant rather than the enum value to keep the audit record human-readable.

This is an O(n) change across all `AuditEvent` construction sites in `tools.rs`.

## Acceptance Criteria

- AC-01: When an MCP client sends `initialize` with `clientInfo.name = "codex-mcp-client"`,
  subsequent tool calls in the same session produce `audit_log` rows where:
  - `agent_attribution = "codex-mcp-client"`
  - `metadata` JSON contains `"client_type":"codex-mcp-client"`

- AC-02: When an MCP client sends `initialize` with an empty `clientInfo.name` (`""`), subsequent
  audit rows have `agent_attribution = ""` and `metadata = '{}'` — no `client_type` key written.

- AC-03: When no rmcp session context is available (STDIO path with no initialize-time capture,
  or a tool call where `client_type_map` has no entry), `agent_attribution = ""` and
  `metadata = '{}'`. Zero-session-id tool calls are unaffected.

- AC-04: The `audit_log` table has all four new columns after migration:
  `credential_type TEXT NOT NULL DEFAULT 'none'`,
  `capability_used TEXT NOT NULL DEFAULT ''`,
  `agent_attribution TEXT NOT NULL DEFAULT ''`,
  `metadata TEXT NOT NULL DEFAULT '{}'`.
  Existing rows receive valid defaults — no data loss.

- AC-05: `AuditEvent` struct has all four new fields with `#[serde(default)]`. `log_audit_event`
  writes all four. `read_audit_event` reads all four back correctly.

- AC-05b: Append-only DDL triggers are installed in the same migration. Subsequent UPDATE or
  DELETE on `audit_log` raises ABORT with the documented error message.

- AC-06: The `initialize` handler override on `UnimatrixServer` does not break the existing
  `get_info()` behavior — the `InitializeResult` returned is identical to the default.

- AC-07: `client_type` propagation is rmcp-session-scoped: two concurrent HTTP sessions with
  different `clientInfo.name` values (e.g. one Codex session and one Gemini session) produce
  correctly attributed audit rows with no cross-session bleed. The `client_type_map` key is the
  `Mcp-Session-Id` UUID — distinct per session.

- AC-08: On stdio transport (no HTTP session), if `clientInfo.name` is available at initialize
  time, it is captured and written to `metadata` for all subsequent tool calls in that connection.

- AC-09: Schema migration bumps the schema version counter. Existing databases migrate without
  data loss.

- AC-10: `client_type` values are truncated at 256 characters before storage. Values exceeding
  this length are logged at WARN level and truncated (not rejected).

## Constraints

**rmcp 0.16.0**: `ServerHandler::initialize` is a provided method with a default implementation.
Overriding it is supported and is the correct extension point. The `InitializeRequestParams`
struct is `#[serde(rename_all = "camelCase")]` — `client_info` maps to JSON `clientInfo`.

**Session ID mismatch**: The rmcp transport-level `Mcp-Session-Id` (an `Arc<str>` UUID managed by
the session manager) is NOT the same as the agent-declared `session_id` tool parameter (used to
key `SessionRegistry` and `AuditContext.session_id`). Bridging these two requires the server to
either (a) key client_type on the rmcp-level session ID and resolve it at tool call time, or (b)
use a server-scoped singleton for the stdio case and a per-rmcp-session map for the HTTP case.
This is the primary architectural decision — see OQ-01.

**Concurrent HTTP sessions**: The streamable HTTP server can have multiple concurrent sessions,
each with a distinct `clientInfo.name` (e.g., one Codex session and one Gemini session active
simultaneously). The implementation MUST be keyed on the session identifier, not a shared mutex
over a single value.

**`AuditEvent.session_id` is a `String` (not `Option<String>`)**: Empty string is the current
sentinel for "no session". The new fields follow the same pattern: `agent_attribution` and
`capability_used` use empty string as sentinel; `credential_type` defaults to `"none"`;
`metadata` defaults to `"{}"`. No nullable fields — `#[serde(default)]` on all four.

**Schema migration scope**: Only `audit_log` needs the new column. The `AuditEvent` struct
lives in `unimatrix-store` — the migration and struct change are confined to that crate.
`unimatrix-server` only changes how it constructs `AuditEvent` instances.

**ASS-050 coordination**: ASS-050 is COMPLETE. VNC-014 delivers the full four-column migration
specified in ASS-050 Section 4. Do not add a `metadata`-only migration first and the other three
columns later — they must land together in one schema version bump. `metadata` is
`NOT NULL DEFAULT '{}'`, not nullable.

**No tool schema changes**: No `#[tool]` struct gains a new field. Agent-facing API is unchanged.

arrives as `params.client_info.name` at the server. Not configurable, not empty. AC-01 applies
for Claude Code sessions: `agent_attribution = "claude-code"`.

**OQ-03 — RESOLVED.** The rmcp-level session ID is accessible via `extensions.get::<Parts>()`
+ `Mcp-Session-Id` header at tool call time. This is rmcp's documented mechanism for HTTP-level
data in tool handlers (`tower.rs:62-69`). Stateless mode: header absent, treat as `""`. See
`product/research/ass-050/FINDINGS-OQ01.md` for full evidence.

**OQ-04 — RESOLVED.** ASS-050 is COMPLETE. Full schema specified: four columns, append-only
triggers, non-nullable `metadata`. VNC-014 delivers this migration in full. See
`product/research/ass-050/FINDINGS.md` Section 4.

**OQ-05 — RESOLVED by OQ-01 resolution.** `client_type` is captured at the connection/rmcp-session
level (keyed on `Mcp-Session-Id`), not the agent-declared `session_id` level. This means
`agent_attribution` is populated for ALL tool calls in a session regardless of whether the caller
passes `session_id` as a tool parameter — the attribution is bound to the rmcp transport session,
not to Unimatrix's logical session concept.
