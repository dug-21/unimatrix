## Scenario 2: Agent Orientation (Subagent Pulls Everything from Unimatrix)

Scrum-master is spawning `ndp-rust-dev` for feature NXS-010. Instead of pasting protocol excerpts into the spawn prompt, it calls `context_briefing`.

### MCP Tool Call (by scrum-master)

```json
{
  "tool": "context_briefing",
  "params": {
    "role": "ndp-rust-dev",
    "task": "implement vector index wrapper with FilterT integration",
    "phase": "implementation",
    "feature": "nxs-010"
  }
}
```

### Server Processing

```
1. lookup(topic="ndp-rust-dev", type="role")
   -> entry:12 "ndp-rust-dev role definition" (duties, boundaries, self-checks)

2. lookup(topic="implementation-protocol", type="protocol")
   -> entry:7 "Implementation Swarm Protocol" (excerpted: agent rules section only)

3. lookup(topic="ndp-rust-dev", type="rule")
   -> entry:25 "Rust workspace conventions"

4. search(query="vector index wrapper FilterT integration", type="knowledge", k=3)
   -> entry:89 "hnsw_rs FilterT closure pattern" (0.91)
   -> entry:92 "HNSW persistence two-file dump" (0.84)
   -> entry:45 "redb spawn_blocking async pattern" (0.79)
```

### Response (compact markdown, ~800 tokens)

```markdown
## Orientation: ndp-rust-dev

### Your Duties
- Implement Rust code following project conventions
- Use anyhow for app errors, thiserror for library errors
- Every public fn gets doc comments and error path tests
[...excerpted from role entry...]

### Protocol (Implementation)
- Return: file paths + test pass/fail + issues (NOT file contents)
- Max 2 validation fix iterations
- Read IMPLEMENTATION-BRIEF.md before starting
[...excerpted from protocol entry...]

### Workspace Rules
- `cargo clippy --workspace -- -D warnings` must pass
- No `unwrap()` in production code
[...from rule entry...]

### Relevant Patterns (3)
1. **hnsw_rs FilterT closure** (0.91): Build closure over `HashSet<usize>` of
   valid data IDs. Pass to `search_filter`. Pre-filtering during traversal.
2. **HNSW persistence** (0.84): Two-file dump (graph + data), NOT atomic.
   Write to temp + rename. Use redb as crash-safe metadata store.
3. **redb async pattern** (0.79): `tokio::task::spawn_blocking` with
   `Arc<Database>`. Clone Arc, move into closure.
```

### Scrum-Master's Spawn Prompt

```
Task(
  subagent_type: "ndp-rust-dev",
  prompt: "You are implementing the vector index wrapper for NXS-010.
    Your agent ID: nxs-010-agent-1-rust-dev

    Context from Unimatrix:
    {paste briefing response above}

    Read: product/features/nxs-010/IMPLEMENTATION-BRIEF.md
    Read: product/features/nxs-010/pseudocode/vector-index.md

    YOUR TASK: Implement VectorIndex wrapper struct...
    Files to create: crates/nexus-core/src/vector/mod.rs, .../index.rs
    RETURN FORMAT: 1. Files modified  2. Tests pass/fail  3. Issues"
)
```

The subagent never calls Unimatrix. All context arrived via the orchestrator-passes-context pattern. `.claude/` files exist on disk but the agent didn't read them -- the briefing contained the relevant excerpts.

## Scenario 3: The 12-Release Retrospective (Automated Process Improvement)

Feature COL-007 took 12 implementation releases due to repeated wave 2 drift. Agents kept building components that didn't integrate because they lacked shared architecture context.

### Retrospective Entries Accumulate

**After release 4** (agent stores via MCP):
```json
{
  "tool": "context_store",
  "params": {
    "content": "Wave 2 agents for COL-007 built incompatible interfaces. Agent-2 used async traits, agent-3 used sync. Neither read ARCHITECTURE.md which specified sync-first with async wrappers.",
    "topic": "implementation-protocol",
    "category": "retrospective",
    "tags": ["protocol-friction", "col-007", "architecture-drift"]
  }
}
```

**After release 8**, similar entry stored. **After release 11**, third entry.

### Cluster Detection (automatic, on third store)

```
context_store triggers retrospective analysis:
  1. Embed new entry
  2. Search: 10 retrospectives for topic="implementation-protocol"
  3. Found 3 entries with similarity > 0.85, all citing "architecture context missing"
  4. Create RETRO_CLUSTERS entry:
     cluster_id: 1
     target_topic: "implementation-protocol"
     entry_ids: [201, 287, 334]
     status: Detected
```

### Correction Proposal (automatic)

System generates a correction to the implementation protocol. The correction targets the specific section: agent spawn template.

```
Original (entry:7, v1, section "Agent spawn prompt template"):
    "Read these files before starting:
    - product/features/{feature-id}/IMPLEMENTATION-BRIEF.md
    - product/features/{feature-id}/pseudocode/{component}.md
    - product/features/{feature-id}/test-plan/{component}.md"

Corrected (entry:7 -> new entry:340, v2):
    "Read these files before starting:
    - product/features/{feature-id}/IMPLEMENTATION-BRIEF.md
    - product/features/{feature-id}/architecture/ARCHITECTURE.md    [ADDED]
    - product/features/{feature-id}/pseudocode/OVERVIEW.md           [ADDED]

Reason: "3 retrospectives (COL-007 releases 4, 8, 11) cite architecture drift
         caused by wave 2 agents not reading ARCHITECTURE.md. Adding mandatory
         architecture and overview files to spawn template."
```

### Result

`.claude/protocols/implementation-protocol.md` now includes ARCHITECTURE.md and OVERVIEW.md in every agent spawn template. The next swarm that runs will read the improved protocol. No human had to diagnose the pattern across 12 releases -- the system detected it, proposed the fix, and the human approved with one keystroke.

## Scenario 4: Protocol Evolution Over 3 Feature Cycles

Tracking the implementation protocol through NXS-008, NXS-009, NXS-010.

### Cycle 1: NXS-008

Protocol at v1 (original seed). Wave 2 runs fine. No retrospective entries. Protocol unchanged.

### Cycle 2: NXS-009

Two agents produce overlapping code because task boundaries were unclear. Retrospective stored:
```
topic: "implementation-protocol", category: "retrospective"
content: "NXS-009 agents 2 and 3 both implemented the query parser.
Task descriptions overlapped. Need explicit file-path ownership per agent."
```

Only 1 retrospective -- below auto-threshold. No correction proposed. But the entry is findable:

```bash
$ unimatrix search "task overlap" --type retrospective
1. NXS-009 task overlap (entry:267) [retrospective, active]
   > Agents 2 and 3 both implemented query parser. Task descriptions overlapped.
```

Human reads this, decides to act early:

```bash
$ unimatrix edit implementation-protocol
# Adds to Step 3c: "Each agent's task MUST list exclusive file paths.
# No two agents modify the same file in the same wave."
# Saves, creating v2
```

```bash
$ unimatrix export
Updated: .claude/protocols/implementation-protocol.md (v1 -> v2)
```

### Cycle 3: NXS-010

Protocol at v2 (with file-path exclusivity rule). Agents work without overlap. But a new issue: agent spawned for `crates/nexus-core/src/vector/mod.rs` also needed to create `crates/nexus-core/src/vector/index.rs` (not in the task list). Agent stopped and reported a blocker instead of creating the file.

Retrospective stored. Similar entry from NXS-011 later. And NXS-012. Auto-threshold hit:

```
Cluster detected: 3 retrospectives citing "rigid file-path exclusivity blocks necessary file creation"
Correction proposed: v2 -> v3
  Change: "exclusive file paths" -> "exclusive file path PREFIXES.
  Agents own a directory subtree, not individual files. May create new files
  within their assigned prefix."
```

```bash
$ unimatrix log implementation-protocol
v1  2026-02-21  seed from .claude/  (original)
v2  2026-03-05  cli:edit            "add file-path exclusivity rule"
v3  2026-03-19  auto:retro-cluster  "relax to prefix-based ownership"

$ unimatrix review
[1/1] implementation-protocol: v2 -> v3
  (a)pprove? a

$ unimatrix export
Updated: .claude/protocols/implementation-protocol.md (v2 -> v3)
```

The protocol has evolved three times: once from seed, once from human edit after reading a single retrospective, and once from automated cluster detection. Each version is tracked with attribution and rationale. The `.claude/` file always reflects the current version. `unimatrix log` provides the full audit trail of why the protocol looks the way it does.
